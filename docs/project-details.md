# Seller Trust Map

Подробное описание дипломного проекта.

## 1. Назначение проекта

`Seller Trust Map` — это сервис для анализа надежности продавцов и проверки достоверности рейтинга карточки товара на маркетплейсах.

Основная задача проекта:

- определить, можно ли доверять продавцу;
- оценить, не выглядит ли рейтинг товара искусственно завышенным;
- показать признаки накрутки отзывов;
- дать пользователю понятный итоговый trust score и объяснение результата.

Проект сделан в формате `backend-first`, потому что основной акцент идет на серверную часть:

- API;
- бизнес-логику;
- антифрод-анализ;
- работу с PostgreSQL;
- кэширование;
- хранение истории проверок;
- расширяемую архитектуру сервиса.

## 2. Общая идея системы

Пользователь открывает карточку товара на маркетплейсе. Далее возможны два сценария:

1. Через браузерное расширение пользователь нажимает кнопку проверки.
2. Через локальный сайт dashboard пользователь вставляет ссылку вручную.

После этого backend:

1. Принимает ссылку товара.
2. Определяет маркетплейс.
3. Извлекает `product id` из URL.
4. Пытается получить данные о товаре и продавце из PostgreSQL.
5. Если данных нет, использует fallback-логику.
6. Пытается дополнительно получить сигналы из HTML страницы товара.
7. Отправляет данные в Python anti-fraud сервис.
8. Вычисляет итоговый `trust score`.
9. Сохраняет результат проверки в историю.
10. Возвращает результат клиенту.

## 4. Архитектура проекта

Проект состоит из нескольких модулей.

### `backend-go`

Основной сервер приложения.

Функции backend:

- маршрутизация API;
- принятие URL товара;
- определение маркетплейса;
- извлечение `product id`;
- обращение к базе данных;
- fallback-генерация данных;
- обращение к Python anti-fraud сервису;
- вычисление trust score;
- сохранение истории проверок;
- выдача данных сайту и расширению.

### `ml-service`

Отдельный сервис для анализа подозрительных отзывов.

Он рассчитывает:

- `fake_review_risk`
- `rating_manipulation_risk`
- `text_similarity_risk`
- `review_burst_risk`

### `browser-extension`

Тонкий клиент.

Расширение:

- берет ссылку текущей вкладки;
- отправляет ее в backend;
- получает результат;
- показывает trust score пользователю.

### `dashboard`

Встроенный сайт, который отдается прямо backend-сервисом.

Он нужен для локальной демонстрации проекта на защите.

Через него можно:

- запускать проверку товара;
- видеть историю своих проверок;
- смотреть trust score;
- смотреть page signals;
- демонстрировать backend-логику без расширения.

## 5. Поддерживаемые маркетплейсы

На текущем этапе проект ориентирован на:

Backend умеет распознавать ссылки этих маркетплейсов и извлекать из них идентификаторы товаров.

## 6. Логика антифрод-проверки

Проверка строится не на одном факторе, а на совокупности сигналов.

### 6.1. Сигналы продавца

Используются метрики продавца:

- общий рейтинг продавца;
- количество заказов;
- доля успешных заказов;
- жалобы;
- возвраты;
- средняя оценка отзывов.

Эти метрики влияют на `seller base score`.

### 6.2. Сигналы товара

Используются метрики карточки товара:

- рейтинг товара;
- число отзывов;
- доля пятизвездочных отзывов;
- доля однозвездочных отзывов.

Эти данные влияют на `product signal score`.

### 6.3. Сигналы накрутки отзывов

Python anti-fraud сервис оценивает:

- повторяемость текста отзывов;
- всплески появления отзывов;
- подозрительно высокую долю непроверенных пятизвездочных отзывов;
- вероятность искусственного завышения рейтинга.

### 6.4. Сигналы страницы товара

В backend реализован page fetcher.

Он пытается:

- скачать HTML карточки товара;
- извлечь `JSON-LD`;
- получить название товара;
- получить рейтинг;
- получить число отзывов;
- найти подозрительные повторяющиеся рекламные блоки.

Это усиливает “реальность” проверки, потому что анализ строится не только по `id`, но и по самой странице товара.

## 7. Формирование итогового trust score

Итоговая оценка находится в диапазоне от `0` до `100`.

Примерная логика:

- оценивается надежность продавца;
- оцениваются сигналы товара;
- вычитается штраф за антифрод-риск;
- строится итоговый `trust score`;
- отдельно рассчитывается `rating authenticity`.

Интерпретация результата:

- `80-100` — низкий риск;
- `60-79` — умеренный риск;
- `0-59` — высокий риск.

## 9. История проверок пользователя

В проекте реализована персональная история.

Для этого используется `client_id`.

Как это работает:

- dashboard генерирует локальный `client_id`;
- расширение тоже генерирует собственный `client_id`;
- этот идентификатор отправляется в backend;
- backend сохраняет проверку с привязкой к пользователю;
- затем можно получить именно “мои проверки”.

Это делает проект ближе к реальному продукту, а не к разовой демо-проверке.

## 10. Локальный dashboard

Для удобной защиты в backend встроен сайт:

- `http://localhost:8080/dashboard`

Через него можно показать:

- ввод ссылки товара;
- запуск проверки;
- trust score;
- authenticity score;
- причины риска;
- сигналы страницы;
- персональную историю проверок.

Это особенно удобно для защиты, потому что весь сценарий работает локально и в одном браузерном окне.

## 11. Браузерное расширение

Расширение реализовано как дополнительный клиент.

Сценарий работы:

1. Пользователь открывает страницу товара.
2. Нажимает на иконку расширения.
3. Нажимает кнопку `Проверить товар`.
4. Расширение берет URL активной вкладки.
5. Отправляет запрос в backend.
6. Получает результат и отображает его во всплывающем окне.

Основная логика остается на backend.

## 12. Локальный запуск

Проект можно запустить через:

```bash
docker compose up --build
```

После запуска доступны:

- backend API: `http://localhost:8080`
- dashboard: `http://localhost:8080/dashboard`
- ML service: `http://localhost:8001`

Также можно использовать локальный PostgreSQL и загрузить схему из `infra/postgres/init.sql`.
